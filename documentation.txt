ResearchPilot — Project Documentation
Hackathon 3.0 Submission

WHAT IS RESEARCHPILOT?

ResearchPilot is an AI assistant that helps people read, search, and understand research
papers and books, and can even write a full research report on its own. It turns a slow,
manual process — finding papers, reading them, pulling out key points, comparing them — into
one connected, guided workflow.

For an edtech company, this maps directly onto real classroom and campus needs: students
doing literature reviews, teachers preparing reading material, and researchers trying to keep
up with new papers. It can sit inside a learning platform as a "research and study companion"
feature, saving learners hours of manual searching and reading.

1. PROBLEM STATEMENT

The problem: reading research papers is slow and repetitive. A student may need to read
20-30 papers just to write one literature review — finding the right papers, reading them,
pulling out key points, and comparing them across papers all takes real time.

Most existing tools only solve one small piece of this — a search tool, or a "chat with one
PDF" tool, or a notes app — never the whole journey. ResearchPilot's angle is to connect the
whole flow in one place: search for papers, chat with one paper or many at once, read and
highlight the actual PDF, save notes, and — most ambitiously — have the AI plan its own
research and write a full cited report by itself.

This was built to be realistic for a hackathon: the team built the core pieces first (reading
PDFs, chatting with AI, searching) and then reused them across every feature, instead of
building each feature from scratch.

2. DATASET & DATA HANDLING

ResearchPilot's dataset is the live research paper corpora of arXiv and Semantic Scholar —
together, tens of millions of academic papers across every field — plus whatever papers and
books a user uploads themselves. Rather than working from a static, downloaded snapshot, the
app queries these sources live through their public APIs, so results are always current and
never go stale:

- Papers and books come from two places: files the user uploads themselves, or live public
  research sources — arXiv and Semantic Scholar.
- Chat answers are generated by AI, but always grounded in the actual text of the paper or
  book the user is asking about — not guessed from the AI's general memory.
- The AI research-report feature uses a real autonomous web-research AI that searches the
  open web live at the moment the question is asked, and is instructed to rely only on
  peer-reviewed, academic sources (journals, arXiv, recognized research publications) rather
  than blogs or random web pages.

Cleaning and preparing the data happens automatically on upload: a paper or book's PDF text
is pulled out, split into clean pieces, and — where possible — tagged with which chapter or
section each piece belongs to. This means the AI answers from the right part of a document
instead of a jumbled mix of pages, and long books can be searched chapter-by-chapter instead
of as one giant blob of text.

3. DATABASE DESIGN

ResearchPilot uses MongoDB Atlas, a modern, flexible cloud database, for everything —
including the actual PDF and image files, which are stored safely inside the database itself
rather than on a temporary disk. This means nothing is lost when the app restarts or is
updated, which matters a lot for a live, deployed product.

Two kinds of data live side by side:
- Everyday records: papers, folders, chats, notes, user accounts, and highlights.
- Search-ready data: every paper and book is also broken into small pieces and converted into
  a special searchable format (a "vector index") that lets the AI find the single most
  relevant passage in seconds, similar to how a smart search engine works, rather than just
  matching exact keywords.

This is a live, real-time structure, not a static one: as soon as a user uploads a paper or
sends a chat message, it's immediately available for search and future conversations. A
NoSQL, document-style database was chosen deliberately, since research papers, chats, and
notes naturally vary in shape and don't fit neatly into rigid spreadsheet-style tables — and
because it comes with built-in support for the AI search feature described above.

4. SYSTEM ARCHITECTURE 

The system is split cleanly into a backend (the "engine" that talks to the AI, the database,
and outside research sources) and a frontend (the interface the user actually sees and
clicks through).

The backend is organized around a handful of specialized AI agents, each with one clear job:
- AI Search — searches papers and gives a summarized, cited answer.
- Chat with PDF — answers questions using one paper, several papers, or a whole folder of
  saved papers at once.
- Deep Research Report — the most advanced agent. It turns a question into a detailed
  research brief, then hands that brief to a real autonomous web-research AI, which plans its
  own searches, reads sources, and writes a full report with real, verifiable citations —
  without a human guiding each step.
- Book Chat — a reading assistant for long books. Instead of only searching the current page,
  it's given a search tool and decides for itself when to use it, so a question about one
  chapter can automatically pull in facts from a completely different chapter.

This agent-based design adds real value rather than needless complexity: each agent can be
tested and improved on its own, and letting the AI plan and search for itself produces far
better answers on open-ended questions and long documents than a single fixed search ever
could.

The system is also modular by design — shared building blocks (like the chat window, the
paper-reading view, and the "which papers should I search" picker) are reused across every
feature, so new capabilities can be added without rewriting the app.

5. FRONTEND, BACKEND & API INTEGRATION

The frontend is a clean, responsive web app that works on both desktop and mobile, with a
light and dark mode, an in-browser PDF reader with text highlighting, and chat answers that
appear live, word by word, as the AI writes them — instead of a long silent wait.

The backend is a well-structured web service with a clear, predictable set of addresses for
every action (for example: list papers, add a highlight, send a chat message), using standard
success and error signals so the frontend — or any other tool — always knows exactly what
happened. It also protects itself from accidentally doing the same expensive AI work twice by
mistake (for example, if a user's connection briefly drops and retries), and comes with
built-in, auto-generated API documentation for developers.

A simple login system gates access to the app, so it can't be used by just anyone who finds
the link.

6. LIVE DASHBOARD & VISUALIZATION

ResearchPilot's "dashboard" isn't a numbers-and-charts dashboard — it's a live research
command center. A user picks an AI helper, types a question, and watches live, plain-language
progress updates as the AI works (for example: "searching the web...", "writing your
report..."), instead of staring at a blank loading spinner. Once ready, papers, chats, and
generated reports are neatly organized into dedicated pages ("My Chats," "My Notebooks," "My
Books") so nothing gets lost — everything done in the past is easy to find again later.

The experience is intentionally simple: one text box, one button, and clear status messages
that any non-technical user can follow at a glance.

7. DEPLOYMENT

ResearchPilot is fully deployed and live, not just running on a local machine: the interface
runs on Vercel, the backend engine runs on Render, and the database runs on MongoDB Atlas.
Because all data — including uploaded files — lives in the cloud database rather than on a
temporary disk, the live version stays stable and keeps all its data across restarts and
updates. Clear, step-by-step setup and deployment instructions are included so the project can
be run locally or redeployed by anyone.

WHY THIS MATTERS

Research and studying are slow, repetitive work. ResearchPilot removes the boring parts —
searching, re-reading, summarizing — so a student or researcher can spend their time actually
thinking, not scrolling. For an edtech company, this is a ready-made "research companion"
capability that could plug directly into an existing learning platform.
